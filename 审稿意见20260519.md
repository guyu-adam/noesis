## 二、致命问题（必须修）

### 1. Φ 公式不统一 —— 最严重

`iit.py` 中 `phi_proxy()` 和 `phi_collaborative()` 使用**不同的计算公式**：

| 模式 | 公式 |
|------|------|
| 竞争/随机/无广播 | `0.40·MI_int + 0.35·EI + 0.25·TE` |
| 协作/混合 | `0.35·coalition_MI + 0.30·workspace_MI + 0.25·EI + 0.10·diversity` |

协作 Φ 多了一个 diversity bonus 项，权重也不同。这相当于用两把不同刻度尺子量完说 A > B。审稿人一旦对照代码，直接拒稿。

**修复**：所有实验模式必须使用**同一个 Φ 公式**，或者严格论证为什么需要不同度量（很难论证通）。建议方案：用统一的 `phi_collaborative` 但非协作模式传入空 coalition，此时 `coalition_mi` 和 `diversity` 自然为 0，等价退化到 `phi_proxy`。

### 2. 实验数据是占位符

论文 `cgwt_paper.tex` 中明确标注：
```
%%% TABLE PLACEHOLDER — fill with actual results after experiment completes
```

当前表格数字非常可疑：
- 协作 vs 竞争的 Mann-Whitney U = 576.0，这是 n=24 的理论最大值，意味着两组完全无重叠 —— 真实实验中几乎不可能
- 多个不同比较得出完全相同的 U 值 (576.0)，统计上极度异常
- 竞争模式的 `Mean ΔΦ = 0.2659` vs 随机 `0.2612` 差异过小，但协作 `1.57` 又大了 6 倍

**修复**：运行真实实验获取真实数据，替换所有占位符。真实数据出现不显著或不完美的结果是正常的。

### 3. 样本量不足

- 每模式 24 个观测（8 stimuli × 3 cycles）
- Mann-Whitney U 在 n < 30 时统计效力不足
- 未提供 power analysis

**修复**：stimuli 至少扩展到 15-20 个，cycles per stimulus 至少 5。或者加 power analysis 说明为什么当前样本量足够。

### 4. LLM agent → 神经群体的桥接论证缺失

用 qwen3:4b 作为神经元群体的代理，论文只在 discussion 中简单提了一句。IIT 领域审稿人对 "language model = neural population" 这种类比会非常挑剔。

**修复**：
- 在 Methods 部分增加专门段落论证 LLM agent 与生物神经元群体的对应关系
- 引用 Phua (2025) 和更多 computational neuroscience 文献
- 明确说明这是**计算模型**而非生物模型，限制断言的强度
- 加多模型敏感性分析

### 5. 单一作者 + 独立研究

`Independent Research; dhuojian@gmail.com`。MDPI Entropy 虽然审稿标准低于顶刊，但独立研究者身份仍会引来额外 scrutiny。

**修复**：找一个有学术 affiliation 的合作者作为通讯作者，或者至少提前把 preprint 挂上 arXiv 建立学术记录。

## 三、严重问题（强烈建议修）

### 6. H4 零结果被轻描淡写

竞争模式 vs 随机模式的 ΔΦ 差异不显著（p = 0.481）。这意味着 **GWT 的核心机制（竞争性选择）相比随机选择没有增加 Φ**。这是一个很大的发现，但在论文中被一笔带过。

**修复**：把 H4 的 null result 作为重要发现展开讨论。这恰好支持了 CGWT 的论点——仅靠 competition 不够，需要 coalition。

### 7. 缺少消融实验

无法判断哪个组件贡献了 Φ 提升：
- 是 coalition 结构本身？还是 diversity 项？还是 world model？
- 关掉 world model 只用 Jaccard similarity 会怎样？
- coalition size 从 2 变到 3 会怎样？
- 关掉 consensus threshold 会怎样？

**修复**：增加消融实验部分——每次关掉一个组件，测量 ΔΦ 变化。

### 7. 权重硬编码无论证

Φ 公式中所有权重（0.40, 0.35, 0.25, 0.10）都是硬编码，论文中没有任何敏感度分析或选择依据。

**修复**：至少做权重扫描（weight sweep），展示主要结论对权重选择不敏感。或者从原理推导权重。

### 9. 只测了一个 LLM

只用 qwen3:4b。如果换 llama3.2:3b 或 mistral:7b 结论是否仍成立？

**修复**：至少用 2 个不同规模的模型重复实验，展示结论的模型鲁棒性。

## 五、世界模型代码审查 (`world_model.py`)

逐行审查发现 9 个问题，其中 3 个会导致错误行为。

### Bug 1: `consensus_concepts` 内存泄漏 + 性能退化

```python
# world_model.py:86
self.consensus_concepts.update(new_consensus)  # Counter 只增不减
```

每个 cycle 都向 Counter 追加新 consensus token，但**从不删除过期条目**。跑 100 cycles 后 Counter 内有数千个 key，`get_consensus_score()` 每次调用都要遍历全部。旧 cycle 的 token 持续污染后续 scoring。

**修**：加修剪或衰减机制。简单方案：每 N 个 cycle 清理出现次数 < 3 的 token。更好方案：用时间衰减的 weighted Counter。

### Bug 2: Hybrid 模式在标准 GlobalWorkspace 上 WorldModel 空转

```python
# experiment.py:126-129
world_model = getattr(cws, 'world_model', None)  # GlobalWorkspace 没有此属性
if world_model is None:
    from world_model import WorldModel
    world_model = WorldModel()  # 空的！每个 cycle 新建！
```

标准 `GlobalWorkspace` 从未初始化 `world_model` 属性。每次 hybrid cycle 都 new 一个空的 WorldModel——无历史、无 consensus concepts、无 agent prediction 记录。`get_consensus_score()` 和 `get_prediction_error()` 永远返回 0.5。World model 在此路径上完全失效。

触发条件：`POST /experiment/run` + `mode=hybrid`（全局 workspace 是 `GlobalWorkspace`）。

`run_comparison_experiment()` 为 hybrid/collaborative 模式显式使用 `CollaborativeWorkspace`，所以 `/experiment/compare` 不受影响。但单 cycle API 受影响。

**修**：方案 A — `GlobalWorkspace.__init__` 也初始化 WorldModel。方案 B — API 层拒绝不兼容的 mode + workspace 组合。推荐 A。

### Bug 3: 无线程安全

```python
# world_model.py — 全类无一锁
def update(self, stimulus, proposals):
    self.stimulus_history.append(stimulus)      # list append 非原子
    self.agent_predictions[name].append(content)  # defaultdict 非原子
    self.consensus_concepts.update(new_consensus) # Counter update 非原子
```

Flask 以多线程模式运行。两个并发请求同时调 `update()` 时，`consensus_concepts` 和 `agent_predictions` 可能状态损坏。

注意：`CollaborativeWorkspace.collaborative_broadcast()` 在持有 `self._lock` 时调用 `self.world_model.update()`，那条路径安全。但 `experiment.py:180` 的 `workspace.world_model.update(stimulus, proposals)` 无锁。

**修**：给 WorldModel 加 `threading.Lock`，所有 public 方法加锁。

### 设计问题 4: `world_embedding` 死代码

```python
# world_model.py:98-100 — 每次 update 都计算
self.world_embedding = _average_embedding(
    [content for content in proposals.values() if content]
)
```

`world_embedding` 只在 `summary()` 里报告维度数 (`len(self.world_embedding)`)。没有任何决策逻辑消费它。纯浪费 CPU（每次 update 都要做 Counter + TF 归一化 + 平均）。

**修**：要么删掉 `_average_embedding` 和相关代码，要么让 `get_consensus_score()` 真正用它做语义相似度（但这需要换用 Ollama embeddings API）。

### 设计问题 5-6: 首 cycle 选择退化

第一个 cycle 中所有 agent 的 `consensus_score` 都是 0.5（空世界模型），`prediction_error` 都是 0.5（无历史）。唯一区分度来自 intensity（谁的输出 token 多）。这意味着首 cycle 的选择几乎是随机的。

这本身在理论上说得通（无 prior 时平等），但如果 stimuli 太少而首 cycle 占比大，会引入噪声。论文应提及这一点。

### 设计问题 7: Consensus threshold 硬编码 + 跨 coalition size 不一致

```python
consensus_threshold = n_agents * 0.6
```

2 agents → threshold = 1.2 → token 必须两个 agent 都说才算 consensus（等价于 Jaccard intersection）。3 agents → threshold = 1.8 → 出现 2 次即可，相对宽松。coalition size 从 2 变 3 时，behavior 会变，但代码没注释。论文只测了 size=2，后续消融实验如果改 size 会引入 confounding。

### 设计问题 8: `_average_embedding` 名不副实

函数名和属性名叫 "embedding"，但实际只是 bag-of-words TF 平均，跟 `SemanticMemory` 用的 Ollama `nomic-embed-text` 无任何关系。读代码的人会误以为这是语义级别的表示。

### 设计问题 9: `set(consensus_concepts.keys())` 冗余

```python
# world_model.py:120
relevant = tokens & set(self.consensus_concepts.keys())
```

`Counter.keys()` 返回 dict_keys view，与 `set` 做交集时会隐式转换。`set()` 包装是多余的，每次都创建新 set 对象。

## 六、真实实验数据采集方案

### 6.1 实验目标

120 cycle 是最低门槛（8 stimuli × 3 cycles × 5 modes），建议保守做到 **200+ cycles**：

| 参数 | 当前 | 建议 |
|------|------|------|
| stimuli 数量 | 8 | 20+ |
| cycles per stimulus | 3 | 5 |
| 模型 | qwen3:4b | qwen3:4b + llama3.2:3b |
| 总 cycles | 120 | 500+ |

### 6.2 数据字段

每个 cycle 保存以下字段到 `experiments/<timestamp>/cycles.jsonl`：

```json
{
  "cycle_id": 1,
  "mode": "competitive",
  "stimulus": "What is the relationship between...",
  "phi_before": 0.123,
  "phi_after": 0.456,
  "phi_delta": 0.333,
  "fisher_trace": 0.012,
  "complexity": 0.234,
  "proposals": {"perceptor": "...", "reasoner": "...", "evaluator": "..."},
  "winner": "reasoner",
  "coalition": [],
  "broadcast": "...",
  "broadcasted": true,
  "attention_score": 1.23,
  "ignition_threshold": 1.05,
  "narrative": "...",
  "model": "qwen3:4b",
  "timestamp": "2026-05-19T12:00:00Z"
}
```

### 6.3 自动保存

在 `noesis.py` 和 `experiment.py` 中：
- `/experiment/run` 和 `/experiment/compare` 结束后自动写入 `experiments/` 目录
- 文件名含时间戳，防止覆盖
- 同时生成汇总 `summary.json`（按 mode 聚合的 mean/std）

### 6.4 数据检查

跑完后做 sanity check，以下是预期模式（如果真实数据不符合，比符合更有意思）：

1. `phi_after >= phi_before` 应在 70%+ 的 cycle 成立
2. `phi_delta` 分布应为正偏态（少量 cycle 贡献大幅提升）
3. collaborative/hybrid 的 `phi_delta` 分布高于 competitive/random
4. `no_broadcast` 的 `phi_delta` 应接近 0
5. `complexity` 与 `phi` 应呈中等正相关（r ≈ 0.3-0.5）
6. broadcasted=True 的 cycle 应有更高 `phi_after`
